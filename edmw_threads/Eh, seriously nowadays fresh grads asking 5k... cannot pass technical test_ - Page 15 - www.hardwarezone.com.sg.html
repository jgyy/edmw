asking cause this is a valid scenario i ran into before.
First if you are already authenticated in step 1, and a cookie is returned by the server, after authentication. The server could have a server-side session, where the SESSION ID is stored in the cookie, and there is a CSRF token created and stored into the server-side session.

I don't see it is necessary for step (3) to generate a CSRF token. I assume what you are describing in step 3 is generate a CSRF token to be eventually compared with the one embedded at the server side (The storage method at the server side doesn't matter).

Usually the corresponding CSRF token is also responded into the same HTML page generated, which could be a form parameter, or a javascript variable.

Obviously without the cookie, step 4 will never works properly without an authenticated session. Just the cookie alone would allow the server-side to retrieved the server-side session for principal identification.

The CSRF returned in the next form submission or next ajax POST either in the request body or request query would be compared against the server-side session CSRF, and check of expiry(if any), repeated use, timestamp differences, etc

That would be how I tackle the problem

PS: I am answering the question having you are the developer, are you asking from a 3rd party perspective where you are trying to intercept or scrape ?

asking cause this is a valid scenario i ran into before.
I will answer you this question using a perspective of a 3rd party.

A) The simplest method is simply use a headless browser. Something like puppeteer will do. Because you have control over the browser. You can inject anything you want into the page at javascript level to extract any kind of information that can be found in the DOM. As for the login flow and CSRF, you don't even need to deal with it. Just command the headless browser via the web driver to perform the steps as if you are using selenium.

B) Using JMeter. I know JMeter is a performance testing tool, but it is also a very good simulated web client, less the heavy lifting of a browser performing the DOM rendering. Hence it terms of performance, it is better than technique (A). Using this method, you can pickout Javascript codes found inside the HTML response and execute(eval) it. Obviously evaluating JS using JMeter would not have the context of the DOM, so it will fail depending on how the function works. What you want to know is where exactly is the important part of generating the CSRF at the JS side. That could be easily achieve by setting breakpoints in your browser developer console to inspect the behaviour of the JS, and discover how the CSRF is generated dynamically at the web client end.

I have used both techniques A and B across my career, especially technique B for performance testing on websites that uses CSRF all the time
